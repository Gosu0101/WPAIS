# 기능 개발 7단계 워크플로우

복잡한 기능 개발 시 수동으로 적용하는 체계적인 개발 프로세스입니다.

## 워크플로우 개요

```
Phase 1: Discovery (발견)
    ↓
Phase 2: Codebase Exploration (탐색)
    ↓
Phase 3: Clarifying Questions (질문)
    ↓
Phase 4: Architecture Design (설계)
    ↓
Phase 5: Implementation (구현)
    ↓
Phase 6: Quality Review (검토)
    ↓
Phase 7: Summary (요약)
```

---

## Phase 1: Discovery (발견)

**목표**: 무엇을 만들어야 하는지 이해

**수행 작업**:
- 기능 요청이 불명확하면 명확화
- 해결하려는 문제 파악
- 제약조건과 요구사항 식별
- 이해한 내용 요약 및 확인

**질문 예시**:
```
- 이 기능이 해결하려는 문제는?
- 성능 요구사항이 있나요?
- 선호하는 솔루션이 있나요?
```

---

## Phase 2: Codebase Exploration (탐색)

**목표**: 관련 기존 코드와 패턴 이해

**수행 작업**:
- 유사 기능 분석
- 아키텍처 패턴 파악
- 핵심 파일 식별

**탐색 관점**:
1. "유사한 기능 찾아서 구현 추적"
2. "해당 영역의 아키텍처와 추상화 매핑"
3. "관련 기능의 현재 구현 분석"

**출력 형식**:
```
발견된 유사 기능:
- [기능명] (경로): 사용 패턴 설명

핵심 파일:
- src/xxx/xxx.ts:45 - 핵심 로직
- src/xxx/xxx.ts:12 - 미들웨어
```

---

## Phase 3: Clarifying Questions (질문)

**목표**: 모든 모호함 해소

**확인 항목**:
- 엣지 케이스
- 에러 핸들링
- 통합 포인트
- 하위 호환성
- 성능 요구사항

**질문 형식**:
```
아키텍처 설계 전 확인이 필요합니다:

1. [항목]: [구체적 질문]
2. [항목]: [구체적 질문]
...
```

**중요**: 이 단계에서 모든 모호함을 해소해야 설계 시작 가능

---

## Phase 4: Architecture Design (설계)

**목표**: 여러 구현 접근법 설계

**접근법 유형**:

### 접근법 1: 최소 변경
- 가장 작은 변경, 최대 재사용
- 장점: 빠름, 낮은 위험
- 단점: 결합도 증가, 테스트 어려움

### 접근법 2: 클린 아키텍처
- 유지보수성, 우아한 추상화
- 장점: 깔끔한 분리, 테스트 용이
- 단점: 더 많은 파일, 리팩토링 필요

### 접근법 3: 실용적 균형
- 속도 + 품질 균형
- 장점: 복잡성과 깔끔함의 균형
- 단점: 일부 결합 유지

**출력 형식**:
```
3가지 접근법을 설계했습니다:

접근법 1: [이름]
- 설명
- 장점/단점

권장: 접근법 N - [이유]

어떤 접근법을 사용하시겠습니까?
```

---

## Phase 5: Implementation (구현)

**목표**: 기능 구현

**수행 작업**:
- 명시적 승인 후 시작
- 이전 단계에서 식별된 파일 읽기
- 선택된 아키텍처 따라 구현
- 코드베이스 컨벤션 엄격히 준수
- 진행 상황 추적

**규칙**:
- 승인 후에만 구현 시작
- Phase 2에서 발견된 패턴 따르기
- Phase 4에서 설계된 아키텍처 사용

---

## Phase 6: Quality Review (검토)

**목표**: 코드가 단순하고, DRY하고, 우아하고, 정확한지 확인

**검토 관점**:
1. **단순성/DRY/우아함**: 코드 품질과 유지보수성
2. **버그/정확성**: 기능적 정확성과 로직 에러
3. **컨벤션/추상화**: 프로젝트 표준과 패턴

**출력 형식**:
```
코드 리뷰 결과:

높은 우선순위 이슈:
1. [파일:라인] - 설명

중간 우선순위:
1. [파일:라인] - 설명

어떻게 하시겠습니까?
- 지금 수정
- 나중에 수정
- 현재 상태로 진행
```

---

## Phase 7: Summary (요약)

**목표**: 완료된 작업 문서화

**출력 형식**:
```
기능 완료: [기능명]

구현 내용:
- [항목 1]
- [항목 2]

핵심 결정:
- [결정 1]: [이유]

수정된 파일:
- [파일 경로] (신규/수정)

다음 단계 제안:
- [제안 1]
- [제안 2]
```

---

## 사용 시점

**사용 권장**:
- 여러 파일에 걸친 새 기능
- 아키텍처 결정이 필요한 기능
- 기존 코드와의 복잡한 통합
- 요구사항이 다소 불명확한 기능

**사용 비권장**:
- 한 줄 버그 수정
- 사소한 변경
- 잘 정의된 단순 작업
- 긴급 핫픽스
